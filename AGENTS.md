# DLProjects Agent Constraints

本文件约束仓库根目录下的全部内容。后续在本项目中新增、修改或组织代码与文档时，默认遵守以下规则。

## 项目定位

- 本项目是一个“教学导向 + 可落地实战”的深度学习示例仓库。
- 任何新增内容都应同时服务于“帮助理解”和“能够实际使用”两个目标，避免只讲概念、不落代码，也避免只堆实现、不利于学习。

## 约束规则

### 1. 教学表达优先清晰

- 实现应尽量简洁，只保留完成目标所需的必要复杂度。
- 注释应少而关键，优先解释核心思路、关键步骤、输入输出含义和为什么这样设计。
- 不要添加大量“代码翻译式”注释；注释必须帮助读者更快理解实现重点。
- 示例、脚本和说明文档应让读者能够快速看懂、快速运行、快速定位关键逻辑。

### 2. 理论内容必须服务实战

- 本项目中的理论、方法和算法说明，必须能映射到当前仓库中的实际实现、训练流程、评估方式或工程模块。
- 如果新增理论说明，必须同步体现其在项目中的实际用途，例如：模型结构设计、数据处理、训练策略、评估指标或推理部署。
- 只适合作为背景资料、但暂时无法在本项目中落地的内容，不应作为主体实现或主要文档内容引入。

### 3. 新文件必须符合 README 目录规划

- 新建文件或目录前，必须先对照 [readme.md](./readme.md) 中的目录规划，将内容放入对应的业务方向或工程模块。
- 不要随意在根目录新增零散文件，也不要绕开既有模块结构放置内容。
- 如果确有必要新增 README 当前未覆盖的目录结构，必须在同一轮改动中同步更新 `readme.md`，保证目录规划与实际仓库结构一致。
- 文件路径和命名应直接体现任务归属，便于教学查找、后续扩展和工程复用。

## 执行要求

- 在开始实现前，先检查方案是否同时满足“表达清晰、具备实战价值、目录位置正确”三项要求。
- 如果用户请求与以上规则冲突，应先指出冲突点，再优先给出满足本项目约束的替代实现方案。
- 在说明改动时，优先说明该改动的教学价值和实战价值。
